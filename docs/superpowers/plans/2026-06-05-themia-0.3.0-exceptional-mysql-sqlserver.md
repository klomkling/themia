# Themia 0.3.0 — `Themia.Exceptional` MySQL + SqlServer dialects — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Add `Themia.Exceptional.MySql` and `Themia.Exceptional.SqlServer` provider packages so 0.3.0 ships all three relational engines, each proven against a real engine via Testcontainers.

**Architecture:** Extends the merged neutral engine — each new package implements `IExceptionalSqlDialect` (the same 11-member contract the shipped `PostgresExceptionalDialect` implements) plus an `AddThemiaExceptional<Engine>` DI extension that runs the shared FluentMigrator `ExceptionLogMigration`. The engine, capture pipeline, model, and tests are unchanged. The only shared-code change is the migration, which must render per-provider timestamp column types.

**Tech Stack:** .NET 8 + .NET 10, Dapper, FluentMigrator 6.2.0, `MySqlConnector`, `Microsoft.Data.SqlClient`, Testcontainers.MySql / Testcontainers.MsSql.

**Spec:** `docs/superpowers/specs/2026-06-05-themia-0.3.0-exceptional-neutral-core-design.md` (this extends the "MySQL→0.3.1, SqlServer→0.3.2" slice into a single 0.3.0).

**Key decisions:**
- **Explicit per-dialect SQL** (each dialect ships its full SQL strings, like the shipped Postgres dialect). No shared-fragment refactor — the rollup/paging/quoting differ enough per engine that it's not worth re-touching shipped code.
- **Per-engine timestamp types:** Postgres `timestamptz` (unchanged), MySQL `DATETIME(6)`, SqlServer `datetime2`. All store UTC instants (the engine UTC-normalizes). Filter `TemporalFilterDbType`: Postgres `DateTimeOffset` (unchanged), MySQL `DateTime`, SqlServer `DateTime2`.
- **Identifier quoting & dialect quirks:** Postgres `"x"`, MySQL `` `x` ``, SqlServer `[x]`. MySQL can't self-reference a table in an UPDATE subquery (error 1093) → derived-table wrapper. SqlServer paging = `OFFSET/FETCH`; booleans = `bit` (`= 1`), no `ILIKE`.

- [ ] **Step 2: Make `ExceptionLogMigration` render per-provider timestamp types**

The shipped migration uses `.AsDateTimeOffset()` for `CreationDate`/`LastLogDate`/`DeletionDate`, which is correct for Postgres (`timestamptz`) and SqlServer (`datetimeoffset`) but **MySQL has no offset type**, and we want tz-naive UTC columns for MySQL/SqlServer. Replace the single `Up()` body with three provider-specific table creations sharing a private helper, plus shared indexes. Full new file:

```csharp
using FluentMigrator;
using FluentMigrator.Builders.Create.Table;

namespace Themia.Exceptional.Migrations;

/// <summary>Creates the <c>Exceptions</c> table. Timestamp column types are rendered per provider.</summary>
[Migration(202606060001, "Themia.Exceptional: create Exceptions table")]
public sealed class ExceptionLogMigration : Migration
{
    /// <inheritdoc />
    public override void Up()
    {
        // Postgres: timestamptz (engine stores Kind=Utc → maps natively, no legacy switch).
        IfDatabase("postgres").Delegate(() => CreateTable(c => c.AsDateTimeOffset()));
        // MySQL: DATETIME(6) — no offset type; UTC wall-clock by convention.
        IfDatabase("mysql").Delegate(() => CreateTable(c => c.AsCustom("DATETIME(6)")));
        // SqlServer: datetime2 — UTC wall-clock by convention.
        IfDatabase("sqlserver").Delegate(() => CreateTable(c => c.AsDateTime2()));

        Create.Index("IX_Exceptions_App_Hash_Created")
            .OnTable("Exceptions")
            .OnColumn("ApplicationName").Ascending()
            .OnColumn("ErrorHash").Ascending()
            .OnColumn("CreationDate").Ascending();
        Create.Index("IX_Exceptions_DeletionDate")
            .OnTable("Exceptions").OnColumn("DeletionDate").Ascending();
    }

    private void CreateTable(System.Func<ICreateTableColumnAsTypeSyntax, ICreateTableColumnOptionOrWithColumnSyntax> ts)
    {
        var table = Create.Table("Exceptions")
            .WithColumn("Id").AsInt64().PrimaryKey().Identity()
            .WithColumn("Guid").AsGuid().NotNullable()
            .WithColumn("ApplicationName").AsString(256).NotNullable()
            .WithColumn("MachineName").AsString(256).NotNullable()
            .WithColumn("Type").AsString(1000).NotNullable()
            .WithColumn("Source").AsString(500).Nullable()
            .WithColumn("Message").AsString(1000).NotNullable()
            .WithColumn("Detail").AsString(int.MaxValue).NotNullable()
            .WithColumn("RequestBody").AsString(int.MaxValue).Nullable()
            .WithColumn("Host").AsString(512).Nullable()
            .WithColumn("Url").AsString(2000).Nullable()
            .WithColumn("HttpMethod").AsString(16).Nullable()
            .WithColumn("IpAddress").AsString(64).Nullable()
            .WithColumn("StatusCode").AsInt32().Nullable()
            .WithColumn("ErrorHash").AsString(64).NotNullable()
            .WithColumn("DuplicateCount").AsInt32().NotNullable().WithDefaultValue(1)
            .WithColumn("TenantId").AsString(256).Nullable();
        ts(table.WithColumn("CreationDate")).NotNullable();
        ts(table.WithColumn("LastLogDate")).NotNullable();
        ts(table.WithColumn("DeletionDate")).Nullable();
        table.WithColumn("IsProtected").AsBoolean().NotNullable().WithDefaultValue(false);
    }

    /// <inheritdoc />
    public override void Down() => Delete.Table("Exceptions");
}
```

> **VERIFY-AGAINST-REAL-API note:** FluentMigrator 6.2.0's exact fluent types/`IfDatabase(...).Delegate(...)` surface may differ. The Postgres migration is already proven, so the **non-negotiable invariant** is: Postgres still renders `timestamptz` and its 9 existing integration tests stay green. ## Task 2: MySQL dialect + DI

- [ ] **Step 1: Implement `MySqlExceptionalDialect`** (delete the placeholder). Mirror `PostgresExceptionalDialect`, with: backtick quoting, the derived-table rollup wrapper (MySQL error 1093), `LIKE` instead of `ILIKE`, and `TemporalFilterDbType => DbType.DateTime`.

```csharp
using System.Data;
using System.Data.Common;
using MySqlConnector;
using Themia.Exceptional;

namespace Themia.Exceptional.MySql;

/// <summary>MySQL/MariaDB implementation of <see cref="IExceptionalSqlDialect"/> (MySqlConnector).</summary>
public sealed class MySqlExceptionalDialect : IExceptionalSqlDialect
{
    private readonly string connectionString;

    /// <summary>Creates the dialect over <paramref name="connectionString"/>.</summary>
    public MySqlExceptionalDialect(string connectionString) => this.connectionString = connectionString;

    /// <inheritdoc />
    public DbConnection CreateConnection() => new MySqlConnection(connectionString);

    /// <inheritdoc />
    public DbType? TemporalFilterDbType => DbType.DateTime;

    /// <inheritdoc />
    public string RollupSql => """
        UPDATE `Exceptions`
        SET `DuplicateCount` = `DuplicateCount` + 1, `LastLogDate` = @LastLogDate
        WHERE `Id` = (
            SELECT `Id` FROM (
                SELECT `Id` FROM `Exceptions`
                WHERE `ErrorHash` = @ErrorHash AND `ApplicationName` = @ApplicationName
                  AND `DeletionDate` IS NULL AND `CreationDate` >= @RollupSince
                ORDER BY `CreationDate` DESC LIMIT 1
            ) AS `t`);
        """;

    /// <inheritdoc />
    public string InsertSql => """
        INSERT INTO `Exceptions`
        (`Guid`,`ApplicationName`,`MachineName`,`Type`,`Source`,`Message`,`Detail`,`Host`,`Url`,`HttpMethod`,`IpAddress`,`StatusCode`,`ErrorHash`,`DuplicateCount`,`TenantId`,`CreationDate`,`LastLogDate`,`DeletionDate`,`IsProtected`,`RequestBody`)
        VALUES (@Guid,@ApplicationName,@MachineName,@Type,@Source,@Message,@Detail,@Host,@Url,@HttpMethod,@IpAddress,@StatusCode,@ErrorHash,@DuplicateCount,@TenantId,@CreationDate,@LastLogDate,@DeletionDate,@IsProtected,@RequestBody);
        """;

    /// <inheritdoc />
    public string GetByGuidSql => "SELECT * FROM `Exceptions` WHERE `Guid` = @Guid;";

    /// <inheritdoc />
    public string ListSql => """
        SELECT * FROM `Exceptions`
        WHERE (@ApplicationName IS NULL OR `ApplicationName` = @ApplicationName)
          AND (@TenantId IS NULL OR `TenantId` = @TenantId)
          AND (@From IS NULL OR `CreationDate` >= @From)
          AND (@To IS NULL OR `CreationDate` <= @To)
          AND (@Search IS NULL OR `Type` LIKE @Search OR `Message` LIKE @Search)
          AND (@IncludeDeleted OR `DeletionDate` IS NULL)
        ORDER BY `LastLogDate` DESC LIMIT @PageSize OFFSET @Offset;
        """;

    /// <inheritdoc />
    public string CountSql => """
        SELECT COUNT(*) FROM `Exceptions`
        WHERE (@ApplicationName IS NULL OR `ApplicationName` = @ApplicationName)
          AND (@TenantId IS NULL OR `TenantId` = @TenantId)
          AND (@From IS NULL OR `CreationDate` >= @From)
          AND (@To IS NULL OR `CreationDate` <= @To)
          AND (@Search IS NULL OR `Type` LIKE @Search OR `Message` LIKE @Search)
          AND (@IncludeDeleted OR `DeletionDate` IS NULL);
        """;

    /// <inheritdoc />
    public string ProtectSql => "UPDATE `Exceptions` SET `IsProtected` = TRUE WHERE `Guid` = @Guid;";

    /// <inheritdoc />
    public string SoftDeleteSql => "UPDATE `Exceptions` SET `DeletionDate` = @DeletionDate WHERE `Guid` = @Guid AND `DeletionDate` IS NULL;";

    /// <inheritdoc />
    public string HardDeleteSql => "DELETE FROM `Exceptions` WHERE `Guid` = @Guid;";

    /// <inheritdoc />
    public string PurgeSql => "DELETE FROM `Exceptions` WHERE `IsProtected` = FALSE AND `CreationDate` < @OlderThan;";
}
```

## Self-review notes
- **Migration is the one shared-code change** — the invariant is "Postgres stays `timestamptz` and its 9 tests stay green." Verify across all three engines.
- **Dialect SQL parity:** every dialect's `InsertSql` column list must match `ExceptionEntry`/the migration (20 columns incl. `RequestBody`). The engine passes the same param names to all dialects — keep them identical (`@ErrorHash`, `@RollupSince`, `@Offset`, `@PageSize`, `@OlderThan`, `@DeletionDate`, …).
- **`@IncludeDeleted`:** bare boolean for Postgres/MySQL; `= 1` for SqlServer. **Don't unify.**
- **Per-engine timestamp DbType:** Postgres `DateTimeOffset`, MySQL `DateTime`, SqlServer `DateTime2` — each chosen to match its column type; the engine's `ToUtc` coercion handles Kind.
- The GA-driver quirks (Guid format on MySQL, datetime2 binding on SqlServer) are where time goes — resolve in the dialect/test setup, prove with Testcontainers, never weaken a test.
