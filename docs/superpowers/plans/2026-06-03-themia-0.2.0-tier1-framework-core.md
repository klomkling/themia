# Themia 0.2.0 — Tier 1: `Themia.Framework.Core` Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended)
> or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`)
> syntax for tracking.

**Goal:** Port zenity-v2 `Zenity.Framework.Core` into `Themia.Framework.Core` (net10.0) — the DDD core
(entities, value objects, Result/Error, domain events, tenancy context, module system) — by relocation
+ namespace rename, with **no v1 fold** and the module interface renamed to `IThemiaModule`.

**Architecture:** This is the first tier of the framework-core layer (spec:
`docs/superpowers/specs/2026-06-02-themia-0.2.0-framework-rename-design.md`, §3). `Themia.Framework.Core`
is **zenity-v2, canonical** — a near-verbatim port (namespace `Zenity.Framework.Core.* → Themia.Framework.Core.*`,
`IZenityModule → IThemiaModule`). It is **net10.0** (framework TFM policy) and references only
`Microsoft.Extensions.DependencyInjection.Abstractions`. `Themia.Caching` and `Themia.Logging` are
separate plans (independent verbatim ports — they have no Zenity deps).

**Audit result (resolves the spec's deferred fold decision — recorded here, do not re-litigate):**
Neither `Maybe<T>` nor `IDateTimeProvider`/`DateTimeProvider` is consumed by **any** package in the
0.2.0 port set. `Maybe<T>` appears only inside the v1 `Zenity.Core` types being **dropped** (its
`Result`/`IAuditable`/`ISoftDeletable`); `DateTimeProvider` has **zero** usages anywhere
(`grep -rn` across `nuget/zenity/src` + `nuget/zenity-v2/src`, excluding `obj/`/`bin/`). → **Fold
neither** (YAGNI). v2's `Error`/`Result`/`ResultExtensions` are the canonical Result stack; v1's
duplicate `Error`/`Result` stay dropped. Framework.Core ports **no** v1 source.

**Tech Stack:** .NET 10 (`net10.0`); `Microsoft.Extensions.DependencyInjection.Abstractions`; xUnit;
central package management; `PublicAPI.*` surface tracking; `Directory.Build.props` (Nullable,
ImplicitUsings, TreatWarningsAsErrors, GenerateDocumentationFile).

### Transform rules (apply to EVERY ported file)

1. **Namespace declarations:** `namespace Zenity.Framework.Core` → `namespace Themia.Framework.Core`
   (covers all sub-namespaces: `.Primitives`, `.Abstractions.Entities`, `.Abstractions.Events`,
   `.Abstractions.Tenancy`, `.Events`, `.Extensions`, `.Modules`).
2. **Cross-reference usings:** `using Zenity.Framework.Core.*` → `using Themia.Framework.Core.*`.
3. **Module type rename:** every `IZenityModule` identifier → `IThemiaModule` (declaration in
   `Modules/IThemiaModule.cs`, and the reference(s) in `Modules/ModuleDescriptor.cs`).
4. **Remove redundant explicit usings** covered by `ImplicitUsings` (`System`, `System.Collections.Generic`,
   `System.Linq`, `System.Threading`, `System.Threading.Tasks`) — Directory.Build.props enables them
   repo-wide. Keep non-implicit usings (`System.Collections.Concurrent`, `System.Linq.Expressions`,
   `Microsoft.Extensions.DependencyInjection`, `Microsoft.Extensions.DependencyInjection.Extensions`).
5. **No other edits.** Do not "improve" the ported code. Port real framework types only — no demo/benchmark
   fixtures exist in this package, but audit from clean source (`-not -path '*/obj/*' -not -path '*/bin/*'`).
6. **Test inputs:** rename `Zenity.Framework.Core.*` → `Themia.Framework.Core.*` in test `using`s and any
   in-test type references; keep assertions/behavior identical.

**Explicitly NOT ported (audit result):** v1 `Zenity.Core/Types/{Maybe,Error,Result}.cs`,
`Zenity.Core/Providers/DateTimeProvider.cs`, v1 `Abstractions/*` — none are consumed by the 0.2.0 set.

  "code" is relocated v2 source (a port, not new code), so per-task source is referenced by the
  source-of-truth file map rather than reproduced — matching the Tier-0 plan's established pattern.
- **Type consistency:** the only renamed type is `IZenityModule → IThemiaModule` (Task 7), applied to both
  its declaration and `ModuleDescriptor`'s reference; all other types keep their v2 names.
- **Known risk:** `DomainEventDispatcher`/its DI extension are the only types touching the
  `Microsoft.Extensions.DependencyInjection` (non-abstractions) namespace; v2 builds with only the
  Abstractions package referenced, so Task 4 verifies they still resolve. If a missing-type error appears,
  the type is in `Microsoft.Extensions.DependencyInjection.Abstractions` — do not add the full DI package.
- **Deferred (not this plan):** generator incrementality (`.Combine`→`ForAttributeWithMetadataName`),
  THEMIA011 "service type not implemented", `IServiceRegistrationAttribute` dead surface — these are
  **tooling** follow-ups; Framework.Core consumes the generator but does not modify it. Track separately.
