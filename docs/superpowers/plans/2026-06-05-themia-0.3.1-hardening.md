# Themia 0.3.1 — hardening (P0 blocker + P1 framework correctness + P2 cheap wins)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Clear the audited, verified backlog for 0.3.1 — unblock the first real consumer (EzyAssets), fix two framework correctness HIGHs, and sweep the cheap wins.

**Scope (from the 2026-06-05 backlog audit, verified vs `main`):** **P0** = the DI source-gen CS0121 blocker; **P1** = EFCore tenant-source mismatch + Npgsql concurrency token; **P2** = cheap wins across Exceptional / Mediator / tooling. P3 → 0.3.2; P4 deferred.

**Tech stack:** .NET 8 + .NET 10; Roslyn incremental generators (netstandard2.0); EF Core 10 + Npgsql; Dapper; xUnit + Testcontainers.

**Design decisions (locked; surfaced for review):**
- **P0:** make the generated `Themia.Generated.ThemiaServiceRegistrations` **`internal`** (each assembly's registration is its own implementation detail — consumers call their own; internal types don't collide across assemblies). *Rejected:* assembly-unique public names (uglier API).
- **P1a:** `ValidateTenantAccess` (Find) reads the **same tenant source the runtime query filter reads, per strategy** — static `TenantContextAccessor.CurrentTenantId` for `RuntimeTenantAccess`, the injected/constant for `PerTenantModel`. So Find and the filter can never disagree.
- **P1b:** for **Npgsql**, use **`UseXminAsConcurrencyToken()`** (server-maintained system column) instead of `IsRowVersion()` on the `byte[]` (which Npgsql maps to non-server-populated `bytea`). Keep `IsRowVersion()` for other providers. The `byte[] RowVersion` column stays mapped (minimal change); a later cleanup may drop it under Postgres.

**No `<Version>` bump** — cutting 0.3.1 is a separate release PR after merge.

## P0 — DI source-gen CS0121 blocker

### Task 1: Generated DI registration class is `internal`

**Root cause:** `ServiceRegistrationGenerator` emits `Themia.Generated.ThemiaServiceRegistrations` via `ThemiaSourceWriter.OpenClass(...)`, which hardcodes `public` (`src/tooling/Themia.Generators.Abstractions/Emission/ThemiaSourceWriter.cs:47-50`). Every referencing assembly emits the same `public` FQN, so a consumer that references `Themia.Mediator` (uses the gen) AND uses the DI-gen gets **CS0121** (ambiguous `Themia.Generated.ThemiaServiceRegistrations`).

**Files:**
- Modify: `src/tooling/Themia.Generators.Abstractions/Emission/ThemiaSourceWriter.cs`
- Modify: `src/tooling/Themia.SourceGenerator/Generators/ServiceRegistrationGenerator.cs` (and `SourceBuilder.cs` if it owns the class emission)
- Modify: PublicAPI of `Themia.Generators.Abstractions` if `OpenClass` signature changes
- Test: `tests/Themia.SourceGenerator.Tests/` (the e2e compile-gate) + `tests/Themia.DependencyInjection.Tests/`

- [ ] **Step 1: Add an accessibility option to `OpenClass`.** Change `OpenClass(string name, bool isStatic = false, bool isPartial = false)` to accept an accessibility, e.g. add `bool isInternal = false` (or `string accessibility = "public"`). Emit `internal` when requested instead of the hardcoded `public`. Update the XML doc ("Emits a public class…" → accurate). Keep the default `public` so the mediator-gen path is unchanged. Add the new line to `PublicAPI.Unshipped.txt` (the signature changed) via `dotnet format analyzers … --diagnostics RS0016`.

- [ ] **Step 2: Emit the DI registration class as `internal`.** In `ServiceRegistrationGenerator` (the place that calls `OpenClass("ThemiaServiceRegistrations", isStatic: true)`), pass the internal flag so the generated `static class ThemiaServiceRegistrations` and its `AddThemiaServices` method are `internal`. (An `internal static` class makes the extension method internal-effective — exactly the per-assembly scoping we want.)

- [ ] **Step 3: Regression test — the exact CS0121 scenario.** In the e2e compile-gate test, add a case that compiles a consumer assembly which **references an assembly that also used the DI-gen** (simulate `Themia.Mediator`’s generated `Themia.Generated.ThemiaServiceRegistrations`) **and** runs the DI-gen itself. Assert the consumer compilation has **no CS0121** and that calling its own `AddThemiaServices()` resolves. (Before the fix this reproduces CS0121; after, it compiles.) Follow the existing gate's pattern of compiling generated output and asserting `GetDiagnostics()` is error-free.

- [ ] **Step 4: Adjust any test that called the registration cross-assembly.** If a test in a separate assembly invoked `Themia.Generated.ThemiaServiceRegistrations.AddThemiaServices(...)` directly, it must now generate+call within its own compilation (internal). Fix those call sites.

- [ ] **Step 5: Build + tests.** `dotnet build Themia.sln --no-incremental` 0-warning; `dotnet test tests/Themia.SourceGenerator.Tests tests/Themia.DependencyInjection.Tests` green.

- [ ] **Step 6: Commit** `fix(tooling): emit DI registration class as internal to avoid cross-assembly CS0121`.

## P1 — framework correctness (Tier-3)

### Task 2: EFCore `ValidateTenantAccess` reads the same tenant source as the filter

**Root cause:** `ThemiaDbContext.ValidateTenantAccess` uses `CurrentTenantId` = injected `tenantContext?.CurrentTenantId` (`ThemiaDbContext.cs:77,261,307`), while the RuntimeTenantAccess global filter reads static `TenantContextAccessor.CurrentTenantId` (`GetCurrentTenantExpression`, `:511-517`). Under non-standard wiring (no `tenantContext`, or the static accessor mutated after construction) Find and the filter disagree → a Find can return/hide the wrong tenant's row.

**Files:** Modify `src/framework/Themia.Framework.Data.EFCore/ThemiaDbContext.cs`; Test: `tests/Themia.Framework.Data.EFCore.IntegrationTests/` (and unit tests).

- [ ] **Step 1: Add a strategy-aware effective-tenant accessor.** Add a private member that returns the tenant the **filter** uses, by strategy:

```csharp
// The tenant the runtime query filter actually evaluates against — Find must use the SAME source
// per strategy so it can never disagree with the filter.
private TenantId? EffectiveFilterTenantId =>
    TenantIsolationStrategy == TenantIsolationStrategy.RuntimeTenantAccess
        ? TenantContextAccessor.CurrentTenantId   // same source as GetCurrentTenantExpression()
        : CurrentTenantId;                          // PerTenantModel: injected/constant
```

- [ ] **Step 2: Use it in both `ValidateTenantAccess` overloads.** Replace the `var currentTenantId = CurrentTenantId;` reads (`:261`, `:307`) with `EffectiveFilterTenantId`. Leave the rest of the comparison logic unchanged.

- [ ] **Step 3: Test the disagreement is gone.** Add an integration test (Postgres, RuntimeTenantAccess): construct the context, set `TenantContextAccessor.CurrentTenantId` to tenant A, seed a tenant-B row, and assert `Find`/`FindAsync` for the B row returns null (matching the filter), and a Find for an A row succeeds — including the case where the injected `tenantContext` and the static accessor would otherwise differ. Pin it with a comment that Find must mirror the filter.

- [ ] **Step 4: Build + the existing EFCore suites stay green** (`dotnet test tests/Themia.Framework.Data.EFCore.Tests tests/Themia.Framework.Data.EFCore.IntegrationTests`). Commit `fix(efcore): Find reads the same tenant source as the runtime filter (no cross-tenant disagreement)`.

### Task 3: Npgsql optimistic concurrency via `UseXminAsConcurrencyToken`

**Root cause:** `ConfigureConcurrencyToken` (`ThemiaDbContext.cs:453-468`) calls `builder.Property(nameof(IConcurrencyAware.RowVersion)).IsRowVersion()` for all providers. On Npgsql a `byte[]` `IsRowVersion()` maps to `bytea` (client-side, never server-populated) → `SaveChanges` never raises `DbUpdateConcurrencyException`. Npgsql's server-maintained token is the system `xmin` column.

**Files:** Modify `src/framework/Themia.Framework.Data.EFCore/ThemiaDbContext.cs`; Test: `tests/Themia.Framework.Data.EFCore.IntegrationTests/Concurrency/`.

- [ ] **Step 1: Branch the concurrency-token config by provider.** In `ConfigureConcurrencyToken`, for entities implementing `IConcurrencyAware`:

```csharp
if (Database.IsNpgsql())
    builder.UseXminAsConcurrencyToken();   // server-maintained system column; the right token for PostgreSQL
else
    builder.Property(nameof(IConcurrencyAware.RowVersion)).IsRowVersion();
```

`UseXminAsConcurrencyToken()` is the `Npgsql.EntityFrameworkCore.PostgreSQL` entity-type-builder extension (already referenced by this project). `Database.IsNpgsql()` is available in `OnModelCreating`. The `byte[] RowVersion` column stays mapped (a normal column under Postgres) — minimal change; note in a comment that dropping it under Postgres is a later cleanup.

- [ ] **Step 2: Test that a real conflict now throws.** In the integration concurrency tests, add: load the same entity in two contexts, save a change in context 1, then save a conflicting change in context 2 and assert `DbUpdateConcurrencyException` is thrown (today it silently succeeds). The existing concurrency integration test (`ConcurrencyIntegrationTests`) currently asserts model-config only — strengthen it / add this real-conflict case.

- [ ] **Step 3: Build + integration green** (Testcontainers Postgres). Commit `fix(efcore): use UseXminAsConcurrencyToken on Npgsql so optimistic concurrency actually fires`.
