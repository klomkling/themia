# Themia 0.2.0 — Tier 2: `Themia.Services` Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended)
> or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Port a **subset** of zenity-v1 `Zenity.Services` → `Themia.Services` (net10.0): the 4
cross-cutting **marker** interfaces (`IService`/`IDomainService`/`IInfrastructureService`/
`IIntegrationService`) **plus** the 11 cross-cutting **infrastructure-service contracts**
(`IEmailService`/`ISmsService`/`IPushNotificationService`/`IStorageService`/`IReportExportService`/
`ICacheProviderAccessor`/`IBackgroundJobScheduler`/`ISecretsProvider`/`IAuditLogService`/
`ITokenService`/`IEventBus`) and their supporting record types — as forward-seams for future modules
(0.4.0+). **Drop** all business-**domain** contracts, all **integration** (domain-specific) contracts,
and **all** `NotImplemented` stub defaults. Port by relocation + namespace rename + `Zenity → Themia`
rewire onto `Themia.Caching`.

**Architecture:** Spec `docs/superpowers/specs/2026-06-02-themia-0.2.0-framework-rename-design.md` §8a
("`Themia.Services` scope — markers + infra contracts only") and §3 ("`Mediator`/`Services` are ported
and rewired … repointed onto `Themia.Framework.Core` + the tooling attributes + `Themia.Caching`/
`Themia.Logging`"; "Drop all demo/benchmark fixtures"). Tier 2 cross-cutting layer (spec §4), framework
TFM `net10.0`, under `src/framework/`. Depends only on `Themia.Caching` (Tier 1) for the
`IThemiaCacheProvider` surfaced by `ICacheProviderAccessor`.

**Subset-port audit (recorded here — do not re-litigate):** The v1 package has **11 source files**.
`IService`/`IDomainService`/`IInfrastructureService`/`IIntegrationService` are **defined in this
package** (not folded into `Framework.Core` — confirmed: no `IService` marker exists anywhere else in
the Themia tree). `InfrastructureServices.cs` holds **11 generic cross-cutting interfaces** (the spec
named 9; the clean source additionally has `ICacheProviderAccessor` and `IEventBus`, both generic
cross-cutting) → **all kept**. `DomainServices.cs` (11 business interfaces) and `IntegrationServices.cs`
(10 domain-specific integration interfaces) are **dropped** per the framework/app boundary guard. The
three `Default*Services.cs` stub files and the stub-registering body of `ServiceCollectionExtensions.cs`
are **dropped** (demo fixtures). The v1 csproj's `FluentValidation`, `Microsoft.Extensions.Logging`,
`Microsoft.Extensions.Options` references are **dropped** — none of the kept files use them (verified:
kept files reference only `Zenity.Core.Caching` for the cache-provider type). The only real dependency
of the kept set is `Themia.Caching`.

**Tech Stack:** .NET 10 (`net10.0`); `Themia.Caching` (ProjectReference) + transitively its
`Microsoft.Extensions.DependencyInjection`; xUnit; central package management; `PublicAPI.*` surface
tracking; `Directory.Build.props` (Nullable, ImplicitUsings, TreatWarningsAsErrors,
GenerateDocumentationFile). No v1 test suite exists → one DI smoke test is added.

### Transform rules (apply to EVERY ported file)

1. **Namespace declarations:** `namespace Zenity.Services.Abstractions` → `namespace Themia.Services.Abstractions`.

   `namespace Themia.Services`.)

   `using Zenity.Core.Caching;` → `using Themia.Caching;` (the ported `IThemiaCacheProvider` declares
   `namespace Themia.Caching;`), and rename the property type **`IZenityCacheProvider` →
   `IThemiaCacheProvider`** in `ICacheProviderAccessor`.
3. **Remove ImplicitUsings-covered usings** (`System`, `System.Collections.Generic`, `System.IO`,
   `System.Threading`, `System.Threading.Tasks`) — `Directory.Build.props` enables them repo-wide.
   `System.IO` **is** covered by ImplicitUsings → drop it (`Stream` resolves). Keep no other usings
   (after the caching rewire, `InfrastructureServices.cs` keeps only `using Themia.Caching;`).
4. **Drop the domain + integration contract files and all `NotImplemented` stub files** — do not port.
   Port real cross-cutting framework types only. Audit from clean source
   (`find … -not -path '*/obj/*' -not -path '*/bin/*'`).
5. **No other edits.** Do not "improve" the ported XML-doc or members. (The `IDomainService`/
   `IIntegrationService` markers' XML-doc examples reference domain types like `IOrderPricingService`/

### Task 4: Minimal `AddThemiaServices` registration seam (rewritten — no stubs)

**Decision (recommended, justified):** Keep a **documented placeholder** `AddThemiaServices` +
`ThemiaServicesBuilder`. The v1 `AddZenityServices` did exactly two things — call `AddZenityCaching()`
and `TryAdd…` ~30 `NotImplemented` stubs. All stubs are dropped (demo fixtures, §8a: "registers no
throwing stubs — infra contracts have no implementation until their modules ship (0.4.0+)"). With no
default infra implementation (not even for `ICacheProviderAccessor`), there is nothing valid to
register, so the body is a no-op. We **keep** the extension (rather than omit it) because it preserves
the public DI entry-point seam that future infra modules (0.4.0+) hook via the `configure` builder
callback — matching the established `AddThemiaCaching`/`AddThemiaLogging` shape — at the cost of one
small documented file. We **do not** call `AddThemiaCaching()` here: caching is a Tier-1 package the
consumer composes directly; pulling it implicitly through Services would be a hidden side effect with no
Services-level payoff now that the cache-accessor stub is gone.

This is **new code shown in full** (a rewrite, not a verbatim port).

**Files:**
- Create (rewrite): `src/framework/Themia.Services/ServiceCollectionExtensions.cs`

- [ ] **Step 1: Create `ServiceCollectionExtensions.cs`** with the minimal seam:

```csharp
using Microsoft.Extensions.DependencyInjection;

namespace Themia.Services;

/// <summary>
/// Entry point for wiring Themia service defaults. The cross-cutting infrastructure-service
/// contracts in <c>Themia.Services.Abstractions</c> have no default implementation in 0.2.0 — they
/// are forward-seams whose implementations ship with their respective Themia modules (0.4.0+). This
/// extension therefore registers nothing by default; it exists to give consumers and future modules a
/// single, stable place to compose service registrations via the <paramref name="configure"/> callback.
/// </summary>
public static class ServiceCollectionExtensions
{
    /// <summary>
    /// Registers Themia service defaults. No services are registered by default in 0.2.0; pass
    /// <paramref name="configure"/> to add your own implementations of the infrastructure-service
    /// contracts.
    /// </summary>
    /// <param name="services">The service collection to configure.</param>
    /// <param name="configure">Optional callback to register service implementations.</param>
    /// <returns>The same <see cref="IServiceCollection"/> for chaining.</returns>
    /// <exception cref="ArgumentNullException"><paramref name="services"/> is <see langword="null"/>.</exception>
    public static IServiceCollection AddThemiaServices(
        this IServiceCollection services,
        Action<ThemiaServicesBuilder>? configure = null)
    {
        ArgumentNullException.ThrowIfNull(services);

        configure?.Invoke(new ThemiaServicesBuilder(services));

        return services;
    }
}

/// <summary>
/// Provides access to the service collection for registering Themia service implementations when
/// calling <see cref="ServiceCollectionExtensions.AddThemiaServices"/>.
/// </summary>
public sealed class ThemiaServicesBuilder
{
    internal ThemiaServicesBuilder(IServiceCollection services) => Services = services;

    /// <summary>Gets the underlying service collection.</summary>
    public IServiceCollection Services { get; }
}
```
