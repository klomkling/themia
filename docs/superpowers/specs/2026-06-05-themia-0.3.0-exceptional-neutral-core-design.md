# Themia 0.3.0 — `Themia.Exceptional` neutral core + PostgreSQL dialect — Design

> Focused implementation-design for the **0.3.0** release slice. Refines the canonical
> [2026-06-01 Themia Exception Logging design](./2026-06-01-themia-exceptional-design.md) for the
> **neutral-core layer only**. The module (`Themia.Modules.ExceptionLogging`), the dashboard, and the
> deferred Serenity adapter are **out of scope here** — they stay as described in the parent spec.

## Context & scope

Phase 0 (framework rename) is complete and **0.2.0 is published** (13 packages). 0.3.0 begins the
**remaining neutral cores**. Per the release strategy and the user's sequencing decision
(2026-06-05): **Exceptional first** (releases `0.3.0`–`0.3.2`, split by DB dialect *if needed*), then
**Quartz** (`0.3.3`–`0.3.5`).

This spec covers the **first slice**: the neutral engine + the first dialect.

### Audit finding — this is a design-build, not a port

An Explore audit across `ezy-assets`, `Idevs.Net.CoreLib`, and the framework source established:

- ezy-assets persists exceptions only via **`Serilog.Sinks.PostgreSQL`** (`needAutoCreateTable:true`,
  table `ExceptionLogs`) — raw append, **no rollup/dedup, no custom store**.
- **No `StackExchange.Exceptional`** reference anywhere; **no existing custom Dapper exception store.**
- The typed-exception hierarchy + ProblemDetails middleware are **already ported** into the shipped
  `Themia.AspNetCore` (0.1.0) — done, not re-done here.
- The only reusable Dapper reference is ezy-assets' `AuditLogRepository` (Dapper + Npgsql).

Therefore the engine, schema, dialect strategy, and capture pipeline are **greenfield**, built TDD
(like the original `Themia.AspNetCore` 0.1.0 work), with the **schema modeled on StackExchange.
Exceptional's proven `Exceptions` table**.

## Packages (0.3.0)

Both `net8.0;net10.0` (net8 leg mandatory for cross-framework reuse), under `src/neutral/`.

| Package | Contents |
|---|---|
| **`Themia.Exceptional`** | `IExceptionStore` + `ExceptionStoreEngine` (Dapper rollup/insert/query/protect/delete/purge), `IExceptionalSqlDialect`, the FluentMigrator table def, capture pipeline (Serilog sink + HTTP enricher + request-body middleware), options. **No driver, no DB-specific SQL.** |
| **`Themia.Exceptional.PostgreSql`** | `Npgsql` driver + `PostgresExceptionalDialect` + FluentMigrator Postgres runner + `AddThemiaExceptionalPostgres(...)` DI entry. |

**Out of 0.3.0:** `Themia.Modules.ExceptionLogging` (net10 + framework-gated → 0.4.0+), the dashboard
(lives in the module), `Themia.Exceptional.MySql` (0.3.1), `Themia.Exceptional.SqlServer` (0.3.2).

## Decisions resolved (2026-06-05)

1. **Schema creation = FluentMigrator.** One migration def, rendered per provider by FluentMigrator
   (ezy-assets precedent). The neutral core can't use EF Core (that's net10 + framework). The
   `IExceptionalSqlDialect` still owns **runtime** SQL (rollup/insert/query); FluentMigrator owns DDL.
2. **Capture surface = full** — Serilog sink + HTTP enricher + request-body middleware, with the body
   middleware **safe-by-default** (opt-in, off by default; `MaxBodyBytes` cap; field-redaction list;
   never captures auth headers) to honor the no-bodies-in-prod rule.
3. **First dialect = PostgreSQL** (EzyAssets is consumer #1 and Postgres-only). MySQL → 0.3.1,
   SqlServer → 0.3.2, built/released as separate patches **only if** the dialect work warrants it.
4. **Schema modeled on StackExchange.Exceptional's `Exceptions` table** (battle-tested; no source to port).
5. **Sink is synchronous** (single fast round-trip, fault-isolated to Serilog `SelfLog`); host opts into
   `Serilog.Sinks.Async` if it wants batching. Building batching into the sink is rejected — it
   complicates the atomic `ErrorHash` rollup for no proven need (YAGNI).

## Schema — `Exceptions` table

FluentMigrator def, rendered per provider. Columns (modeled on StackExchange.Exceptional):

- `Id` — identity PK.
- `Guid` — per-error id (uuid / uniqueidentifier / native per dialect).
- `ApplicationName`, `MachineName`, `Type`, `Source`, `Message` (truncated), `Host`, `Url`,
  `HttpMethod`, `IpAddress`, `StatusCode`.
- `Detail` — `FullJson` payload (provider max-text), **System.Text.Json**.
- `ErrorHash`, `DuplicateCount`.
- `CreationDate`, `LastLogDate`, `DeletionDate` — UTC.
- `IsProtected`.
- **`TenantId` — nullable**, added now for module forward-compat. The neutral core **stamps** it when
  present in the log-event properties but **never filters** by it (no tenant context in the neutral
  layer). This avoids a later add-column migration when the module lands.

Indexes: `(ApplicationName, ErrorHash, CreationDate)` + `DeletionDate`.

## Engine — `Themia.Exceptional`

`ExceptionStoreEngine` (Dapper), all DB-specific SQL behind `IExceptionalSqlDialect`:

- **Rollup-then-insert by `ErrorHash` within `RollupPeriod`** (default **10 min**): atomic
  UPDATE…RETURNING/SELECT (incrementing `DuplicateCount`, refreshing `LastLogDate`) delegated to the
  dialect; on no match, INSERT.
- `FullJson` store + rehydrate via System.Text.Json.
- **Paged + filterable query** (date / app / tenant / free-text) for the future dashboard — not just a
  capped `GetAllErrors`.
- Protect, soft-delete, hard-delete, purge (by age).
- **One connection per method** (analyzer-clean; no in-method swallow-and-rethrow).
- `IExceptionalSqlDialect` surface: connection factory, the rollup SQL, the GUID-param strategy, the
  UTC-now expression, provider max-text type, and the paged-query SQL.

### Options

`ExceptionalOptions` — `ApplicationName`, `RollupPeriod` (default 10 min), enricher scrub-key list,
request-body-middleware toggle + `MaxBodyBytes` + redaction list. Validate on startup
(`ValidateOnStart`) — fail fast on missing connection string / application name.

## Capture pipeline (safe-by-default)

- **`ExceptionalSerilogSink`** — filter `Exception != null && Level >= Error` → `ExceptionStoreEngine`.
  Synchronous emit; **fault-isolated to Serilog `SelfLog`** (a store failure never breaks the app).
  Plugs into the **existing `Themia.Logging` Serilog pipeline** — does **not** create a competing logger.
- **`HttpContextEnricher`** — stamps url / method / host / ip / tenant onto the log event; **scrubs
  `Cookie` / `Authorization`** + configurable extra keys.
- **`RequestBodyLoggingMiddleware`** — **opt-in, off by default**; `MaxBodyBytes` cap; field-redaction
  list; never captures auth headers.

## Provider package — `Themia.Exceptional.PostgreSql`

- Driver: `Npgsql`.
- `PostgresExceptionalDialect`: rollup via
  `WITH updated AS (UPDATE "Exceptions" SET … WHERE "Id"=(SELECT "Id" … ORDER BY … LIMIT 1) RETURNING "Guid") SELECT …`;
  `now() at time zone 'utc'`; native `Guid`/`uuid`; `text` for `FullJson`.
- FluentMigrator Postgres runner.
- `AddThemiaExceptionalPostgres(connectionString, configure?)` — registers store + dialect + sink +
  enricher (+ body middleware if enabled) and runs the migration at startup.

## TFM, standards, testing

- TFM **`net8.0;net10.0`** for both packages.
- PublicAPI tracking (`PublicAPI.Shipped/Unshipped.txt`), full XML docs, analyzer-clean (THEMIA rules),
  `TreatWarningsAsErrors`. Clean `--no-incremental` build reports 0 warnings (RS0016 clean).
- **Testcontainers PostgreSQL** contract tests: insert, rollup/dedup by `ErrorHash`, GUID round-trip,
  `FullJson` rehydrate, soft/hard delete, protect, count, paged+filtered query, migration applies.
- **Unit tests**: enricher scrub (`Cookie`/`Authorization` removed), sink filter (level/exception gate),
  body-middleware redaction + size-cap + off-by-default, `Truncate` helper, options validation.
- **TDD design-build**: failing test → implement → pass → commit; subagent-driven execution + per-area
  spec & code-quality review, mirroring Phase-0 tiers.

## Release sequencing

`0.3.0` = `Themia.Exceptional` + `Themia.Exceptional.PostgreSql` end-to-end. Single-version monorepo:
bump `<Version>`→`0.3.0` in `Directory.Build.props` + promote CHANGELOG `[Unreleased]→[0.3.0]` → merge
→ `release.yml` tags `v0.3.0`, OIDC-publishes **all** packages at 0.3.0, creates the GitHub Release.
Then **assess**: `.MySql` (0.3.1) and `.SqlServer` (0.3.2) ship as separate patches only if their
dialect work warrants its own release; otherwise they ride a later bump.

> **CI reminder (durable, from 0.2.0):** never `dotnet pack --no-build` a solution containing Roslyn
> analyzer/source-generator packages — pack re-invokes Build under `NoBuild=true` → NETSDK1085. The
> release workflow already dropped `--no-build` (PR #47); keep it that way.
