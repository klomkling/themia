## 1. Goal & scope

Land the **framework core + cross-cutting + tooling** layer into the proven Themia harness,
published as a single **0.2.0** milestone. About **12 packages** are assembled from **three
existing source trees** — zenity-v2, zenity-v1, and `Idevs.Net.CoreLib` — renamed to `Themia.*`,
with all ported tests green on their target frameworks, a clean build (no `RS0016`
undocumented-public-API diagnostics), and release through the existing OIDC Trusted-Publishing
pipeline that shipped 0.1.0.

This is **not** a mechanical rename: zenity-v2 supplies the framework `Core`/`Data.EFCore`/
`AspNetCore`, while the cross-cutting packages (`MultiTenancy`, `Mediator`, `Caching`, `Logging`,
`Services`) come from the **older** zenity-v1 tree and must be reconciled onto v2's richer core
(§3). The tooling family additionally merges Idevs' Roslyn generators/analyzers with zenity-v1's
mediator source-generator.

### Source trees (paths, for the implementer)

- **zenity-v2** — `Packages/nuget/zenity-v2/src/framework/Zenity.Framework.{Core,Data.EFCore,AspNetCore}`
  (the rewrite; canonical for framework Core/Data/AspNetCore).
- **zenity-v1** — `Packages/nuget/zenity/src/{Zenity.Mediator,Zenity.Core.Caching,Zenity.Core.Logging,
  Zenity.Services,Zenity.MultiTenancy,Zenity.Primitives,Zenity.SourceGenerator}` (cross-cutting + mediator gen).
- **Idevs** — `Idevs.Net.CoreLib/src/{Idevs.Net.CoreLib.Generators,.Generators.Abstractions,.CodeFixes}`
  (DI source-gen + analyzers + code fixes).

## 3. Reconciliation rules (the v1 ↔ v2 merge)

The single architectural decision driving the cross-cutting ports:

- **`Themia.Framework.Core` is zenity-v2, canonical.** v2's `Framework.Core` is the richer DDD core
  (`Entity`, `ValueObject`, `AuditableEntity`, `Error`, `ResultExtensions`, domain-event
  infrastructure, `ITenantContext`/`TenantContextAccessor`, module system). It wins all overlaps.
- **Fold in only the v1 `Core` types v2 genuinely lacks.** The implementer audits v1
  `Zenity.Core/Types` (`Maybe`, `Result`, `Error`) and `Providers`/`Abstractions`; candidates to
  fold are `Maybe` and `IDateTimeProvider`/`DateTimeProvider`. **Drop** v1's duplicate `Error`/
  `Result` in favor of v2's `Error` + `ResultExtensions`. The audit result is recorded in the Tier 1
  plan; do not drag duplicate "core" concepts into the tree.
- **v1 `Primitives` DI marker attributes → `Themia.Generators.Abstractions`** (tooling), **not**
  `Framework.Core`. (`Primitives` is only `Attributes/LifetimeAttributes.cs` +
  `Attributes/ServiceRegistrationAttributes.cs`.)
- **`Caching` / `Logging` port verbatim** — namespace rename only; they have **no** Zenity project
  dependencies (only external: Redis/MessagePack, Serilog, `Microsoft.Extensions.*`).
- **`Mediator` / `Services` are ported and rewired** — their thin `Zenity.Primitives` /
  `Zenity.Core` usings are repointed onto `Themia.Framework.Core` + the tooling attributes +
  `Themia.Caching` / `Themia.Logging`. (Observed coupling is small: a handful of `using` statements,
  dominated by `Zenity.Core.Caching` / `Zenity.Core.Logging` / `Zenity.Core.Context`.)
- **Drop all demo/benchmark fixtures.** Port real framework types only. The per-tier audits MUST
  enumerate types from **clean source only** (`find … -not -path '*/obj/*' -not -path '*/bin/*'`):
  broad greps over a package directory pick up source-generator output and build artifacts under
  `obj/` (e.g. sample `OrderHandler`/`OrderService`/`CacheService` types appear in *every* v1
  package's `obj/`), which are **not** shippable package source. Treat any LOC/type counts in the
  brainstorm/review as indicative, not authoritative.

### 3a. MultiTenancy reconciliation (the largest overlap — explicit)

`Themia.MultiTenancy` is **not** a clean v1 port: v2 already owns a tenancy model, and tenant
isolation is the framework's security-critical invariant, so this merge is bigger than Mediator's and
is the **Tier 2 plan's headline reconciliation**.

- **v2's `ITenantContext` / `TenantContext` / `TenantContextAccessor` / `ITenantEntity` are
  canonical.** The `Themia.Framework.Data.EFCore` global query filter + audit key off `ITenantEntity`
  / `ITenantContext` (per architecture Data §) — this is the path that actually enforces isolation, so
  it wins.
- **v1's value is its resolution stack, which v2 lacks** — `ITenantResolutionStrategy`
  (`Header`/`Path`/`Default`), `ITenantStore` (`InMemory`/`Cached`/`Dapper`), `ITenantResolver`,
  `MultiTenancyBuilder`/`Options`. Port these, but **rewire them to populate v2's `TenantContext`**
  (not v1's `TenantInfo`/`ITenantAccessor`), so resolution feeds the same context the data layer
  filters on. Drop v1's competing `ITenantAccessor`/`TenantInfo` in favor of v2's accessor unless the
  Tier 2 audit finds a capability v2 lacks.
- **`TenantResolutionMiddleware` collides by name in both trees** (v1 MultiTenancy and v2
  `Framework.AspNetCore/Tenancy`). Collapse to **one** middleware — keep v2's wiring as the host,
  driven by v1's resolver/strategies. Do not ship two.
- **`DapperTenantStore`** reads the *tenant catalog* (not tenant-scoped rows), so it is arguably
  outside the architecture's read-only Dapper-hatch restriction (Data §) — but the Tier 2 plan must
  classify it explicitly (sanctioned catalog read vs. route through EF) rather than port raw Dapper
  silently.

`Themia.Mediator` also lifts the ezy-assets pipeline behaviors (`LoggingBehavior`, `TenantBehavior`,
`ValidationBehavior` + `ISkipTenantValidation`) per architecture §C **only if** they already exist in
the zenity-v1 mediator; otherwise that lift is deferred — 0.2.0 ports what exists, it does not invent.

### 3b. Tooling reconciliation (SourceGenerator merge + analyzer subset)

Both v1 (`Zenity.SourceGenerator`) and Idevs (`.Generators`) ship a **DI service-registration
generator**, and the 4 analyzers live **inside** the Idevs `.Generators` project. Resolution:

- **Idevs DI-gen is canonical, and is the ONLY generator in Tier 0.** `IdevsServiceRegistrationGenerator`
  (~521 LOC, on the `Generators.Abstractions` scanner/writer/`ServiceTypeResolver`/`LifetimeResolver`
  infra) wins. **Drop** v1's simpler `ServiceRegistrationGenerator` (~160 LOC).
- **The v1 mediator generator moves to Tier 2, not Tier 0** *(correction, 2026-06-02 — discovered during
  Tier 0 execution).* `HandlerDiscovery` matches `Themia.Mediator.Abstractions.IRequestHandler<,>` — a
  **Tier 2** type — so the mediator-gen and its snapshot tests have a forward dependency on
  `Themia.Mediator`. It is ported in the Tier 2 plan alongside `Themia.Mediator`, registered as a second
  pipeline in `Themia.SourceGenerator` then.
- **Canonical DI model = the Idevs model, rebranded, in a new runtime package `Themia.DependencyInjection`**
  *(correction)*. The consumer-facing types are runtime types (they depend on
  `Microsoft.Extensions.DependencyInjection.Abstractions` for `ServiceLifetime`) and therefore must NOT
  live in the analyzer-only `Themia.Generators.Abstractions` (`DevelopmentDependency`). Port from the
  Idevs main lib, renaming `Idevs.ComponentModels → Themia.DependencyInjection` and
  `Idevs.Repositories → Themia.DependencyInjection`:
  - attributes `ScopedAttribute`/`SingletonAttribute`/`TransientAttribute` (each `: IServiceRegistrationAttribute`,
    with `ServiceType?`/`ServiceKey?`/`AllowSelfRegistration`/`Lifetime`) + `IServiceRegistrationAttribute`;
  - marker interfaces `IScopedService`/`ISingletonService`/`ITransientService` (+ generic `<TService>`).
  The v1 `[RegisterAs]`/`[GenerateServiceRegistrations]` attribute model is **not** adopted (the Idevs
  DI-gen does not use it). `Themia.DependencyInjection` ships `net8.0;net10.0` (framework-neutral, broadly
  consumable). The DI-gen matches these by fully-qualified name — repoint its hardcoded
  `global::Idevs.ComponentModels.*Attribute` / `global::Idevs.Repositories.I*Service` and emitted
  namespaces (`Idevs.Generated`/`Idevs.Extensions`/`IdevsServiceRegistrations`) to the `Themia.*` forms.
- **`Themia.Generators.Abstractions` holds generator-author helpers only** (scanner/writer/resolvers/
  diagnostics) — no consumer-facing attributes.
- **Diagnostic IDs `IDEVSGEN → THEMIA`**, preserving numbers: DI-gen `THEMIA001–010`, analyzers
  `THEMIA100–103`. Rename the factory `IdevsDiagnostics → ThemiaDiagnostics` (category `Themia.DI`,
  ID regex `^THEMIA[0-9]{3,}$`) and `DiagnosticIdRange` constants.
- **Analyzer subset for 0.2.0 — port only rules whose target types exist now:**
  - ✅ `THEMIA101` SwallowLogRethrow, ✅ `THEMIA102` SyncOverAsync — generic C# patterns, no Themia-type
    dependency.
  - ⬜ `THEMIA100` MultipleConnectionsWithoutUnitOfWork — needs the data/UoW layer (Tier 3+); **defer**.
  - ⬜ `THEMIA103` ManualSequence — needs `ISequenceProvider` (Phase 3); **defer** (with its code fix).
- **`Themia.Analyzers.CodeFixes` is deferred entirely** — the only provider
  (`ManualSequenceCodeFixProvider`) fixes the deferred `THEMIA103`. Shipping it now would be an empty
  package (YAGNI). It returns in Phase 3 with `THEMIA103`.

## 4. Build sequence — 4 dependency tiers → 4 implementation plans

```
Tier 0  tooling:    Generators.Abstractions → Themia.DependencyInjection (runtime) → SourceGenerator (DI-gen only) → Analyzers
                    (see §3b; mediator-gen → Tier 2; CodeFixes deferred)
Tier 1  core:       Framework.Core · Caching · Logging
Tier 2  cross-cut:  MultiTenancy (see §3a — headline merge) · Mediator · Services
Tier 3  data/web:   Framework.Data.EFCore → Framework.AspNetCore
```

- Lower tiers never depend on higher ones; within a tier the arrow shows intra-tier order.
- **Each tier becomes one implementation plan** under `docs/superpowers/plans/`, produced by the
  `writing-plans` skill. A tier may land as one or several PRs; **every PR to `main` gets a Copilot
  review** (repo convention).
- **The `v0.2.0` tag is cut only after Tier 3 lands.** Single shared version: the release PR bumps
  `<Version>` `0.1.0 → 0.2.0` in `Directory.Build.props` and promotes `CHANGELOG.md`
  `[Unreleased] → 0.2.0`; merging it fires the gated OIDC publish that ships all 12 packages at
  0.2.0 (`Themia.AspNetCore` re-publishes at 0.2.0 too — normal for the shared-version layout).

Rationale for tooling-first: `Themia.Mediator` (Tier 2) cannot compile without `Themia.SourceGenerator`
(Tier 0), and the generator merge is the milestone's highest-risk item (§7) — isolating it first means
everything downstream builds on a proven generator.

## 5. Conventions

- **Renames:** `Zenity.* → Themia.*`; `Zenity.Framework.* → Themia.Framework.*`; types
  `IZenityModule`/`ZenityModuleBase` → `IThemiaModule`/`ThemiaModuleBase`; diagnostic IDs
  `IDEVSGEN1xx → THEMIA1xx`. Update `Zenity.sln` references → `Themia.sln`.
- **TFM per csproj** (not in `Directory.Build.props`): tooling = `netstandard2.0`,
  framework = `net10.0`. The `src/<layer>/` folder signals the target.
- **Public API tracking:** each package ships `PublicAPI.Shipped.txt` / `PublicAPI.Unshipped.txt`;
  a clean `--no-incremental` build reports undocumented public members as `RS0016`.
- **Build hygiene** (from `Directory.Build.props`): `Nullable`, `ImplicitUsings`,
  `TreatWarningsAsErrors=true`, `GenerateDocumentationFile=true`; central package management via
  `Directory.Packages.props`. Treat warnings as build failures.
- **Code rules:** `System.Text.Json` only (never `Newtonsoft.Json`); `ILogger<T>` only (no
  `Console.*`); no double-logging in catch clauses.
- **TDD flow** per the project plan convention: failing test → implement → pass → commit; use the
  superpowers `executing-plans` / `subagent-driven-development` skills the plans call out.

## 7. Risks

- **Source-generator merge (highest risk).** Unifying v1's mediator source-gen with Idevs' DI
  source-gen into one `Themia.SourceGenerator` means reconciling two attribute models and two
  incremental-generator pipelines. Tier 0 isolates and proves it before any consumer depends on it.
- **v2 Core gap audit.** The exact set of v1 `Core` types to fold into `Framework.Core` must be
  pinned in the Tier 1 plan to avoid dragging duplicate `Result`/`Error`/context concepts.
- **Analyzer retargeting.** Some Idevs analyzers (UoW, hand-rolled-sequence) reference types that do
  not exist in 0.2.0 (sequences are Phase 3). Port only rules whose target types exist now; defer the
  rest and note the deferral in the Tier 0 plan.
- **Shared-version blast radius.** Bumping to 0.2.0 republishes `Themia.AspNetCore` too; the release
  PR must confirm the 0.1.0 package is unaffected by the framework additions (it has no framework dep).

- `DapperTenantStore` classification: sanctioned catalog read vs. route through EF (Tier 2 plan).
- Whether the ezy-assets mediator pipeline behaviors are already in zenity-v1 or deferred (Tier 2 plan).
- Which Idevs analyzer rules survive the 0.2.0 type set — **resolved in §3b** (`THEMIA101/102` ship;

### 8a. Tier 2 resolutions (decided 2026-06-04, after a source audit of v1/v2)

The four Tier-2 open items above are resolved as follows; the Tier 2 plans implement them.

- **MultiTenancy model — KEEP v1 `TenantInfo`/`ITenantAccessor`, bridge to v2.** The audit found v2's
  `TenantContext` is **not** a superset of v1's `TenantInfo`: v2 holds only a validated `TenantId` +
  `Source`, whereas v1 carries `Name`/`Environment`/**`ConnectionString`**/`Properties` plus
  resolution-result traceability. Per §3a's own conditional ("drop v1's accessor *unless* the audit
  finds a capability v2 lacks"), v1's model is **kept**. The single ported `TenantResolutionMiddleware`
  populates **both**: `ITenantAccessor.Current = TenantInfo` (rich; carries the per-tenant connection
  string) **and** `TenantContextAccessor.CurrentTenantId = TenantId.From(identifier)` (the bridge that
  the Tier-3 EF global filter keys off). This supports **both** isolation models — shared-DB tenant
  filter *and* DB-per-tenant — and a hybrid, with no rework.
- **DB-per-tenant is a first-class driver (EzyAssets).** The first Themia consumer, EzyAssets, will
  offer agency customers a **separate database per company**. `ITenantAccessor.Current.ConnectionString`
  is therefore a **load-bearing contract**: the Tier-3 `Framework.Data.EFCore` `DbContext` factory reads
  it to construct the per-tenant context. Tier 2 *resolves and carries* the connection string; Tier 3
  *consumes* it to route. The boundary holds — `MultiTenancy` must not depend on EF Core. (EzyAssets is
  Postgres-only, so multi-**engine** dialect work stays deferred to Tier 3 / 0.3.0; the only multi-DB
  concern in Tier 2 is the connection-string seam + a portable catalog query.)
- **Security — redact `TenantInfo.ConnectionString`.** v1's `TenantInfo` is a `sealed record`, so its
  generated `ToString()` prints every property, including the credential-bearing `ConnectionString`.
  As an **[INTRODUCED]** fix over the verbatim port, the Tier 2 plan overrides `ToString()` (and/or
  `[JsonIgnore]`s the field) to redact it, with a test asserting the connection string never appears in
  the string/JSON representation. Honors the no-secrets-in-logs rule.
- **`TenantResolutionMiddleware` collapse — port v1's in Tier 2, complete the collapse in Tier 3.** v1's
  middleware is self-contained (depends only on `Microsoft.AspNetCore.Http`) and ships in
  `Themia.MultiTenancy`. v2's header-only stub lives in `Framework.AspNetCore` (**Tier 3**); the "ship
  only one" rule completes there — Tier 3's `UseZenity`/`UseThemia` rewires to the ported middleware and
  deletes the stub. Carry-forward noted for the Tier 3 plan. (Drop v1's `TenantContext<,>` hydration
  line — that internal type is not ported.)
- **`DapperTenantStore` — sanctioned catalog read, ported as-is, with a portability fix.** It queries
  the tenant **catalog** (`SELECT … FROM {tenants} WHERE identifier = @identifier`), not tenant-scoped
  business rows, so it sits outside the read-only Dapper-hatch restriction. Port as plain Dapper
  (consumer supplies the `IDbConnection` factory). **Drop the `LIMIT 1`** (MySQL/PostgreSQL/SQLite-only
  syntax that SQL Server rejects): `identifier` is unique and `QueryFirstOrDefaultAsync` already takes
  the first row, so the query is portable across all four engines with no dialect machinery.
- **Mediator pipeline behaviors — port the 5 that exist; defer `TenantBehavior`/`ISkipTenantValidation`.**
  The audit confirmed zenity-v1's mediator has `Logging`/`Validation`/`Caching`/`Performance`/`Transaction`
  behaviors but **no** `TenantBehavior` or `ISkipTenantValidation`. Per §3a ("port what exists, do not
  invent"), those two are **deferred**. (`TransactionBehavior` is re-checked during planning — if it
  references a Tier-3 UoW/data abstraction, it defers with it.)
- **Mediator source-generator — fold into `Themia.SourceGenerator` as a 2nd pipeline.** Port v1's
  `MediatorHandlerGenerator` + `HandlerDiscovery`, retargeting the matched FQN to
  `Themia.Mediator.Abstractions.IRequestHandler<,>`, emitted namespace `Zenity.Mediator.Generated →
  Themia.Mediator.Generated`, and moving the `[GenerateMediatorHandlers]` assembly attribute into
  `Themia.Mediator` (runtime) so the generator matches by name. Highest-risk item (touches the shipped
  Tier-0 generator) → gated by its snapshot + e2e compile tests.
- **`Themia.Services` scope — markers + infra contracts only.** Ship the 4 cross-cutting marker
  interfaces (`IService`/`IDomainService`/`IInfrastructureService`/`IIntegrationService`) **plus** the
  cross-cutting infra-service contracts (`IEmailService`/`ISmsService`/`IPushNotificationService`/
  `IStorageService`/`IReportExportService`/`IBackgroundJobScheduler`/`ISecretsProvider`/
  `IAuditLogService`/`ITokenService`) as forward-seams for future modules. **Drop** all concrete
  business-**domain** contracts (Accounting/Invoice/Pricing/Tax/Billing/Inventory/ProductCatalog/…) per
  the boundary guard, and **all** `NotImplemented` stub defaults (demo fixtures). `AddThemiaServices`
  registers no throwing stubs — infra contracts have no implementation until their modules ship (0.4.0+).
